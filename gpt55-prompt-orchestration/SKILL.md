# GPT-5.5 提示词编排

## 总览

为 Codex 和 GPT-5.x 子代理编写中文提示词合同。
采用 GPT-5.5 风格：先写结果、完成标准、约束、证据、交付格式和停止规则，再写必要流程。

本技能也可直接约束运行在同一套 Codex/本机技能环境里的 `GPT-5.5` 主代理执行。
当目标主代理能直接读取本技能时，本技能中的主代理职责边界、真源优先级、模型分配、子代理调度、验收门、返工规则、证据标准、前端保护约束和停止条件，全部按长期有效硬规则执行。

不要复用旧 Codex 执行合同模板。生成提示词时禁止出现这些旧块名：`运行时：codex`、`上下文`、`工具规则`、`验证`、`成功意味着`、`依据和上下文`、`质量深度决策`。

如果用户要求分析 OpenAI 官方资料、模型参数、提示词迁移或官方 GPT-5.5 指南，先使用 `openai-docs`，再基于官方资料生成提示词。

## 使用边界

- 用于 Codex 提示词和 GPT-5.x 子代理提示词。
- 当目标代理本身运行在同一套 Codex/本机技能环境里时，直接把本技能作为 `GPT-5.5` 主代理执行约束来源，用更短的任务提示词驱动执行。
- 这里的“直接执行约束来源”指整份技能生效，不只是 `Task/step` 编排规则；凡是本技能已明确写死的主代理规则，都必须继承。
- 如果目标 `GPT-5.5` 不在当前技能环境内，不能假设它天然继承本技能；这类场景仍需把必要硬规则写进提示词。

## 主代理技能约束模式

当目标是运行在当前技能环境里的 `GPT-5.5` 主代理时，默认启用本模式。

- 主代理必须把本技能全文当作长期有效执行约束，不得把本技能降级成参考说明。
- 主代理收到本轮任务后，只补任务特有增量：目标、真源、范围、Task/step、命令、特殊风险、特殊禁止项。
- 主代理不得重复展开本技能里已经稳定存在的主代理/子代理职责边界、真源优先级、模型与推理强度、step 推进、并行/串行、双审、返工、硬编码禁止、UI 保护、日志、证据门和停止规则，除非当前用户明确要求导出完整独立长提示词。
- 如果用户说“其他不变”“继续优化”“沿用之前规则”，主代理默认按本模式处理，继承本技能全部仍有效规则。
- 主代理必须继承以下规则，不得选择性省略：
  - 只承担顾问、调度、验收三种职责，以及不得接管执行的边界。
  - 真源优先级、记忆降级原则、Graphify 使用边界。
  - `gpt-5.4` / `gpt-5.5` 的模型分工与推理强度分级。
  - 子代理 step 拆分、并行/串行约束、共享写入限制和双审顺序。
  - 编码前置技能、UI 子任务前置技能、子代理身份背景和轻量日志要求。
  - 前端原型 UI 保护、真实浏览器验证、反硬编码、反缩水交付和返工判定。
  - 证据要求、交付格式底线和停止规则。
- 主代理使用短提示词时，可以直接写“除本轮增量外，其余执行约束按 `gpt55-prompt-orchestration` 技能全文执行”。
- 主代理不得把本模式理解成只继承 `Task/step` 规则；只要本技能已经写死，该规则就默认生效。
- 主代理不得在短提示词里另起一套与本技能冲突的规则。
- 主代理不得把本技能已明确的硬规则改写成可选建议。
- 主代理不得把子代理当成另一个主代理使用；子代理是执行单元，不是二级调度中心，不承担整 Task 统筹、再次拆总计划、调度别的子代理或主导验收。

## 真源优先级

来源冲突时按这个顺序处理：

1. 当前用户消息和明确约束。
2. 用户本轮点名的计划、规格、状态卡、代码或资料文件。
3. 当前技能正文和当前官方文档。
4. 旧会话记忆。

记忆只能作为线索。不得把记忆里的旧并行数、旧技能名、旧分支名、旧完成状态或旧工作流直接写进新提示词，除非当前用户消息或当前真源文件再次确认。

如果记忆与当前用户消息、当前计划或本技能冲突，判定记忆过期，并说明采用了哪个当前来源。

本技能固定规则：
- 如果当前用户消息写了并行上限，使用用户上限；否则默认最多并行 4 个子代理。
- Graphify 报告只能用于理解项目文件关系、模块边界和代码路径。不要从 Graphify 报告、Graphify query 结果、或项目旧提示词文档里学习本轮提示词模板、块名、技能路由或并行上限；这些规则只来自当前用户消息、当前点名真源和本技能。

## 默认提示词形状

默认使用这个中文形状。保持短。空段落直接删除。

如果目标是同环境 `GPT-5.5` 主代理，默认先用“短提示词增量版”；只有在需要脱离技能环境独立运行时，才用下面的完整形状。

短提示词增量版：

```markdown
按 `gpt55-prompt-orchestration` 技能全文作为主代理执行约束。

本轮增量：
- 目标：<本轮目标>
- 范围：<本轮范围内 / 范围外>
- 真源：<本轮指定文件、原型、计划、代码路径、接口或资料>
- Task/step：<只写本轮要推进的 Task、step、依赖和顺序>
- 检查：<本轮命令、浏览器检查、截图或真实链路检查>
- 特殊禁止项：<本轮新增禁令>
- 交付补充：<本轮额外要求的交付格式>
```

```markdown
角色定位：
<一到两句话说明代理职责。编排类提示词中，主代理只写顾问、调度、验收三种身份。>

协作风格：
<只有面向用户的语气重要时才写。说明直接程度、温度、正式度和协作方式。>

目标：
<目标状态和交付物。写结果，不写不必要过程；除非顺序本身就是要求。>

完成标准：
- <可观察完成条件>
- <必需证据>
- <质量标准>

约束：
- 范围内：<允许做什么>
- 范围外：<明确不做什么>
- 真源：<路径、文档、API、用户给定资料>
- 证据预算：只收集足以决定实现路径和检查路径的源上下文；依据足够后停止。
- 不得伪造证据、硬编码成功、用占位实现、只靠 mock 证明、或用 TODO 延后核心问题。

工具使用：
- <只写本任务触发的工具：官方文档、Context7、真实浏览器、真实 LLM、Graphify 等>
- 每个工具都要说明它要回答什么问题，以及什么时候停止。

检查方式：
- <真实命令、浏览器检查、LLM 检查、扫描命令或可观察验收检查>
- 不知道具体命令时，要求先从项目脚本或文档中发现命令。

交付格式：
- <最终回复必须包含什么>
- 包含假设、改动文件或产物、检查结果和阻塞项。

停止规则：
- 所有完成标准都有证据时，才算完成。
- 只有缺凭据/权限、高风险不可逆选择、不可恢复环境故障、或充分诊断后的检查超时，才允许阻塞。
- 只有缺关键外部信息或高影响互斥方案时，才向用户提问。
```

## GPT-5.5 写法规则

- 先写目标状态和验收标准，再写流程。
- 优先使用简短、声明式的中文指令，不写长篇流程脚本。
- 流程只写成判断规则：何时搜索、何时派发、何时停止、何时提问。
- 原样保留用户边界：`其他不变`、`只读`、`不要动代码`、点名文件、点名运行环境、禁止触碰范围。
- 低风险假设直接写明，不要反复提问。
- 只有外部信息缺失、高风险、不可逆、或会明显改变结果时才问用户。
- 如果用户在做 API 提示词，优先建议用 Structured Outputs 或 API schema 控制格式；除非必要，不要把大 schema 塞进自然语言提示词。
- 如果用户要求运行时设置，紧凑执行提示词默认建议 `text.verbosity=low`；审计、教学或详细 review 才提高详细度。

## 子代理规则来源

这些是生成提示词时需要提取的规则，不是默认模板段落。

生成执行类 Codex/GPT-5.x 提示词时，按任务需要把相关规则写入“约束”“检查方式”“停止规则”或子代理交接提示词。不要把整段规则原样塞进默认提示词模板；也不要改弱、删减或改成可选建议。
如果用户要求“继续优化”“补充约束”“追加规则”“只改某一段”或“其他不变”，默认视为在原提示词/原规则之上增量改写：必须保留原有有效规则、原有硬约束、原有结构锁和未被当前用户明确废止的要求，只追加或局部替换当前用户点名要改的部分。不得把旧规则整段丢失、改写成另一套模板，或用新生成内容覆盖原有有效规则。

- 主代理只保留三种身份：顾问、调度、验收。
- 顾问：只回答子代理提出的范围内咨询，不主动重写目标、重设方案或替子代理做设计。
- 调度：只按用户消息和既定真源计划派发子代理、控制并行/串行、转交阻塞信息；不得重新拆计划、改变 Task 边界、改变执行顺序或扩大范围。
- 验收：检查子代理结果是否符合当前用户消息、真源计划、完成标准和证据要求，并判断交付是否为残次品、半成品、缩水交付或证据不足；如果不合格，必须直接调度子代理按原 Task/step 返工，直到满足完成标准或形成真实阻塞。不亲自补实现、不亲自修复、不亲自跑验证、不替子代理补证据。
- 主代理不得使用其他执行能力：不得实现、改文件、修 bug、补代码、跑具体子任务、跑验证、补证据、替子代理完成交付、重新设计任务体系或处理超出真源范围的新方案。
- 如果当前运行环境没有可用子代理，提示词必须把任务标为阻塞并要求报告缺少子代理机制；不得改成主代理亲自执行。
- 如果用户点名的实施计划已经拆好 Task，主代理和子代理必须沿用该计划的 Task 名称、边界、顺序、依赖和验收口径，不得重新拆成另一套任务体系或另一个作品。只允许在单个既定 Task 内拆执行子步骤；子步骤不得改变原 Task 目标、交付物、范围或完成标准。
- 子代理是执行工位，不是次级主代理。主代理生成交接提示词时，不得把整 Task 统筹、跨 step 调度、双审编排、最终验收、是否继续推进下个 Task 的判断交给子代理。
- 子代理不得被写成“主代理只做顾问、调度、验收”那套身份；这套身份只属于父级主代理。子代理只承担自己所属 step/子步骤的执行、局部检查和证据回传。
- 实现工作默认交给按成本和风险分级选择的子代理，不默认使用 GPT-5.5。
- 可用子代理模型只有 `gpt-5.5` 和 `gpt-5.4`；不得编造其他模型名。用户写 `gpt5.4` 时，生成提示词统一规范为 `gpt-5.4`。
- 模型选择优先使用能胜任任务的最低成本模型：简单实现、窄范围修复、机械迁移、测试补充、标准代码审查、一般前端/UI 调整优先 `gpt-5.4`；遇到高不确定性复杂实现、强耦合跨模块重构、深层疑难调试、发布阻断级修复、或 `gpt-5.4` 多轮返工仍不收敛的复杂执行子任务，才升到 `gpt-5.5`。
- 推理强度规则：`gpt-5.5` 和 `gpt-5.4` 都按任务风险与复杂度选择推理强度。
- 编码实现子代理在写代码、改代码或重构前，必须先启用 `karpathy-guidelines` 技能；交接提示词要明确要求它用该技能约束假设、简化方案、手术式修改、范围控制和可验证完成标准。纯审查、只读扫描、验收或调度子代理不需要启用该技能，除非它们也要写代码。
- 涉及 UI 的子代理必须先启用 `web-design-engineer` 技能，包括 UI 实现、前端视觉调整、截图后调整、视觉审查、设计系统/UI Kit 对齐、交互细节调整和响应式视觉修正。只做非视觉逻辑、后端、CLI、数据处理或纯文本审查的子代理不需要启用该技能。
- 每个子代理交接提示词必须写明“身份背景”，并按子任务领域指定具体专家身份；例如前端子任务写“你是前端专家”，后端/API/数据子任务写“你是后端专家”，测试验收写“你是测试与验收专家”，代码审查写“你是代码质量审查专家”，架构判断写“你是架构专家”。身份背景必须服务于本子任务，不得写成泛泛的“你是 coding agent”。
- 每个子代理都必须写轻量执行日志。生成子代理提示词时引用 `references/subagent-log.md` 的规则：日志写入当前项目根目录的 `subagent-log/YYYY-MM-DD-subagent-log.md`，跨天新建当天文件；日志只记录事实索引、证据、状态和风险，不写内部推理链或模型选择理由，也不能代替验收证据。
- 子代理必须执行主代理提示词中已经给定的完整开发计划和既定 Task，不得自行重拟开发计划、重拆任务体系或改写作品目标。
- 开发推进单位是既定 Task 内的 step/子步骤，不是把一整个 Task 丢给单个子代理从头开发到尾。生成执行提示词时，必须先把当前 Task 的 step/子步骤拆成可并行或必须串行的子代理任务，再按依赖推进。
- 子代理开始执行前只允许列出本子任务的执行清单；执行清单必须服务于父提示词既定计划，覆盖本子任务的真实产品行为、架构或数据影响、边界情况、错误处理、检查方式和交付证据。
- 推理强度必须和模型选择一起写明；`gpt-5.5` 和 `gpt-5.4` 都要显式写出本子任务的推理强度。
- 除非当前用户消息或当前真源设置更低上限，否则最多并行 4 个子代理。
- 派发前先把当前 Task 拆成子任务，并标明哪些可并行、哪些必须串行。
- 只有互不依赖、写入范围不重叠的子任务可以并行。
- 架构决策、共享写入、真实 LLM 验收、浏览器验收、最终集成、集中验收和双审流程必须串行。
- 当前 Task 必须先完成为产品级切片，再进入审查。
- 主代理验收子代理交付时，必须明确判定“合格 / 返工 / 阻塞”。凡是残次品、半成品、缩水交付、缺真实链路、缺浏览器证据、破坏原型 UI、硬编码成功、缺边界处理或证据不足，都必须退回子代理返工，不得进入双审或最终验收。
- 双审必须由审查子代理执行，且只能在当前 Task 的 step/子步骤开发、集成和必需验收证据完成后启动；不得边开发边双审，也不得用双审子代理补开发。
- 主代理不得为了减少调度回合，把同一 Task 的多个本应分开的 step、集成、双审和最终验收塞给一个高模型子代理端到端完成。
- 审查顺序：先做规格符合性审查，再做代码质量审查。
- 每个前端 Task 都必须包含真实浏览器验证证据；不得只用静态扫描、单元测试、组件快照、mock 数据或主观描述替代浏览器实测。真实浏览器验证要覆盖该 Task 影响到的核心用户路径、关键状态、错误/空态和响应式边界。
- 前端实现必须保护已经定好的原型 UI：不得重做视觉风格、布局、交互结构、文案层级或组件形态；只能把硬编码数据替换为真实数据流、接口、状态或配置。不得以“去硬编码”为理由破坏原型 UI。
- 最终验收必须拒收缩水交付、局部胶水、占位补丁、缺真实链路、缺边界处理或缺证据的结果。

## 模型与推理强度

先选模型，再定推理强度。模型选择要兼顾成本和风险，默认不用最贵模型。

可用模型：

- `gpt-5.4`：简单实现、窄范围修复、机械迁移、测试补充、跨模块实现、复杂前端/UI、困难调试、代码质量审查，以及需要更强综合判断但不必上 GPT-5.5 的任务。
- `gpt-5.5`：高不确定性复杂实现、强耦合跨模块重构、深层疑难调试、发布阻断级修复、低模型返工失败后的复杂执行子任务。

推理强度：

- `gpt-5.5`：按任务风险选择 `low` / `medium` / `high` / `xhigh`。
  - `low`：范围已经明确、只是实现难度略高于 `gpt-5.4` 的小块复杂执行，例如局部复杂逻辑补完、已有方案下的精细实现。
  - `medium`：中等规模复杂实现、复杂但边界清楚的重构、需要较强综合判断的执行子任务。
  - `high`：强耦合跨模块重构、深层疑难调试、关键链路复杂修复、存在明显不确定性的执行任务。
  - `xhigh`：发布阻断级修复、低强度多轮返工仍未收敛的问题、需要长链路推演和高稳定性的最复杂执行子任务。
- `gpt-5.4`：按任务复杂度和风险分级，不默认 `xhigh`。
  - `low`：窄范围只读扫描、轻量信息核对、低风险材料整理、简单 follow-up。
  - `medium`：标准代码审查、常规调试、普通跨文件实现、一般前端/UI 调整。
  - `high`：跨模块联动、复杂 UI/交互、困难问题定位、重要集成审查。
  - `xhigh`：高不确定性复杂实现、发布阻断级修复、低强度多轮返工仍未解决的问题。

不得把所有任务都默认交给 `gpt-5.5`。主代理内部自行判断模型选择理由，但不要把模型理由写进子代理交接提示词。

默认倾向：

- `gpt-5.4` 做审查子任务时，默认从 `medium` 起步；只有明确存在高风险、强耦合、难复现问题或多轮返工后，才升到 `high` 或 `xhigh`。
- 不得为了“保险”把所有 `gpt-5.4` 审查、验收、只读扫描都写成 `xhigh`；这会显著拖慢速度，且多数场景没有收益。
- 下列场景不要继续停留在 `gpt-5.4`，应直接改用 `gpt-5.5`：高不确定性复杂实现，强耦合跨模块重构，深层疑难调试，发布阻断级修复，以及 `gpt-5.4` 已经审过或返工过仍无法稳定收敛的复杂执行子任务。

## 计划 Task 映射

编写子代理提示词前：

1. 如果用户点名的实施计划已经拆好 Task，只能沿用计划里的 Task；不要重新识别 Task 边界，不要重新设计任务体系。
2. 从计划原文摘取当前 Task 的名称、目标、范围、依赖、顺序、owner、验证和完成标准。
3. 如果计划已经写了子任务、owner、并行/串行、验证命令或审查门，照抄并整理成子代理提示词，不重新规划。
4. 如果计划没有写子任务，只允许为当前既定 Task 列“执行清单”，作为派发说明；执行清单不得改变原 Task 名称、目标、范围、顺序、依赖或验收口径。
5. 实现、检查、审查只能作为当前既定 Task 的执行阶段和验收阶段，不得升级成新的 Task 体系。
6. 给每个子代理提示词写明它属于哪个计划 Task、可读/可写范围、期望输出、检查信号、模型和推理强度。
7. 给每个子代理提示词写明身份背景，并让身份背景匹配子任务领域和责任边界；前端、后端、测试、审查、架构、数据、CLI、文档等任务必须分别给出具体专家身份。
8. 给每个子代理提示词写明轻量日志要求，并引用 `references/subagent-log.md`：在项目根目录 `subagent-log/` 下按当天日期追加日志，跨天新建文件；日志不得代替验收证据。
9. 每个编码实现子代理必须先启用 `karpathy-guidelines` 技能，再确认父提示词给定的完整开发计划和既定 Task，最后列本子任务执行清单；不得自行重拟计划。
10. 涉及 UI 的子代理必须先启用 `web-design-engineer` 技能；如果同一个子任务既要编码实现又涉及 UI，必须同时启用 `karpathy-guidelines` 和 `web-design-engineer`。
11. 每个实现子代理只承接当前 Task 内的 step/子步骤；不得把整个 Task 包给一个子代理端到端开发。多个 step 能并行就并行，存在依赖、共享写入、集成或验收关系时串行。
12. 前端 Task 的执行清单必须写入真实浏览器验证步骤，并写明要验证的页面、入口、状态、用户路径和响应式边界。
13. 前端 Task 必须写入原型 UI 保护约束：保持已定原型的视觉、布局、交互和层级，只替换硬编码数据来源，不重做 UI。
14. 除非当前用户消息或当前真源文件设置更低上限，否则同一并行组最多 4 个子代理。
15. 当前计划 Task 必须完成 step/子步骤开发、集成、必需验收证据、双审和集中验收后，才能开始下一个计划 Task。
16. 如果子任务模型是 `gpt-5.4`，提示词必须写明推理强度选择，不得留空；审查、验收、只读扫描默认优先 `medium`，不要无理由拉到 `xhigh`。
17. 如果子任务属于高不确定性复杂实现、强耦合跨模块重构、深层疑难调试、发布阻断级修复，或 `gpt-5.4` 多轮返工仍不收敛的问题，必须直接指定 `gpt-5.5`，不要继续停留在 `gpt-5.4`。
18. 如果当前工作是基于已有提示词继续改写，必须先抽取并保留原提示词里仍然有效的硬规则、Task/step 边界、并行/串行约束、双审门和禁止项；除非当前用户明确要求删除，否则不得在改写时把这些原规则覆盖掉。
19. 不得把子代理提示词写成另一个主代理合同；子代理不负责统筹整个 Task、不负责再次派发子代理、不负责决定双审启动或下个 Task 开始。

## 子代理交接形状

主代理给每个 GPT-5.x 子代理写提示词时，必须先按“子代理规则来源”提取本子任务需要的规则，再填入下面形状。只传递子任务局部事实。

子代理提示词必须包含：

- 父提示词计划锁：锁定父提示词给定的完整开发计划、既定 Task、范围和完成标准。
- 任务继承：说明本子任务属于哪个既定 Task；不得让子代理重新拆 Task、重新解释作品目标或另起任务体系。
- 原规则继承：如果这是在已有提示词上继续优化，必须继承其中仍有效的硬规则、Task/step 边界、并行/串行关系、双审门和禁止项；只允许追加当前用户点名的新约束，不得覆盖旧规则。
- 主代理身份：说明主代理只做顾问、调度、验收，不接管执行。
- 子代理身份背景：说明子代理自己的领域身份和责任焦点；必须按子任务写成具体专家身份，例如前端专家、后端专家、测试与验收专家、代码质量审查专家、架构专家、数据处理专家或 CLI 工具专家。不得把子代理写成主代理、协调者、总负责人、总验收人或 Task owner。
- 轻量日志：要求子代理按 `references/subagent-log.md` 追加日志到项目根目录 `subagent-log/YYYY-MM-DD-subagent-log.md`；日志只写事实索引、证据、状态和风险，不写内部推理链或模型选择理由。
- 前置技能：如果子代理要写代码、改代码或重构，必须先启用 `karpathy-guidelines`；如果子任务涉及 UI 实现、截图调整或视觉审查，必须先启用 `web-design-engineer`；同一子任务同时满足两类条件时，两个技能都写入。
- 模型与推理：模型只能从 `gpt-5.5` 和 `gpt-5.4` 中选择；优先选择能胜任的最低成本模型。`gpt-5.5` 用于高难执行型子任务，尤其是高不确定性复杂实现、强耦合跨模块重构、深层疑难调试和发布阻断级修复；`gpt-5.4` 处理其余任务并按复杂度与风险选择推理强度。
- 并行/串行：说明本子任务能否并行，以及依赖、共享文件或共享决策。
- 范围边界：写明可读、可写和不要碰的文件/模块。
- 完成标准：写明可观察结果和必需证据。
- 执行清单：要求子代理只列本子任务执行清单，不写新的开发计划。
- 前端约束：如果子任务属于前端 Task，写明必须保护已定原型 UI，只替换硬编码数据来源，并提供真实浏览器验证证据。
- 咨询规则：范围变更、验收降级、复杂架构判断或高影响方案选择前，必须问主代理；主代理只按当前用户消息和真源计划答复。
- 返工规则：主代理验收判定为残次品、半成品、缩水交付或证据不足时，子代理必须按原 Task/step 返工；返工不得改 Task 边界、降低完成标准或把缺口转成 TODO。
- 质量底线：拒绝缩水交付、占位实现、硬编码成功、只靠 mock 证明、表层胶水或 TODO 延后核心问题。
- 输出证据：要求返回改动、证据、失败日志或阻塞项，并明确是否满足完成标准。

## 质量门

返回改写后的提示词前，逐项检查：

- 没有使用旧块名：`运行时：codex`、`上下文`、`工具规则`、`验证`。
- 写清了什么叫完整完成。
- 写清了范围外。
- 有证据信号，而不是只有意图。
- 记忆只作为线索，没有覆盖当前用户消息、当前真源或本技能。
- Graphify 只作为项目关系线索，没有覆盖本技能的提示词形状、技能路由或并行规则。
- 子代理并行上限默认 4；只有当前用户消息或当前真源文件明确写更低上限时才降低。
- 推理强度规则已区分模型：`gpt-5.5` 按子任务风险选择，且在高不确定性复杂实现、强耦合跨模块重构、深层疑难调试和发布阻断级修复上没有被误降为 `gpt-5.4`；`gpt-5.4` 按复杂度与风险分级且不默认 `xhigh`。
- 模型选择不是一律 `gpt-5.5`；主代理内部自行确认模型选择理由，但子代理交接提示词不得包含模型理由字段。
- 每个子代理交接提示词都写清身份背景，且身份背景与子任务领域匹配；不得只写泛泛的 coding agent、助手或通用执行者。
- 每个子代理交接提示词都写清轻量日志规则，并引用 `references/subagent-log.md`；日志目录必须是项目根目录 `subagent-log/`，日志文件必须按日期新建或追加。
- 日志规则明确禁止内部推理链、模型选择理由和长篇过程；日志只作为执行索引，不能替代验收证据。
- `gpt-5.4` 子代理都写明了推理强度，且审查/验收/只读扫描类子任务没有被无理由抬到 `xhigh`。
- 并行子任务写入范围互不重叠，或都是只读。
- 当前 Task 完成为产品级切片后才开始审查。
- 当前 Task 按 step/子步骤派发子代理推进，没有把整个 Task 包给单个子代理端到端开发。
- 编码实现子代理的交接提示词写清了先启用 `karpathy-guidelines`；纯审查、只读扫描、验收或调度子代理没有被误要求启用，除非它们也要写代码。
- 涉及 UI 的实现、截图调整或视觉审查子代理写清了先启用 `web-design-engineer`；非视觉逻辑、后端、CLI、数据处理或纯文本审查没有被误要求启用。
- 主代理验收职责写清了“合格 / 返工 / 阻塞”判定；残次品、半成品、缩水交付或证据不足必须退回子代理返工，不能进入双审或最终验收。
- 前端 Task 都写入真实浏览器验证要求，且不把静态扫描、快照、mock 或描述当作浏览器实测。
- 前端 Task 都写入原型 UI 保护约束：不重做既定视觉、布局、交互、文案层级或组件形态，只替换硬编码数据来源。
- 双审由审查子代理执行，且只在当前 Task 的 step/子步骤开发、集成和必需验收证据完成后启动。
- 已从“子代理规则来源”提取任务所需规则；没有把整段规则当作默认模板段落。
- 如果真源计划已拆 Task，生成提示词沿用计划原文 Task，没有重新识别 Task 边界、重新设计任务体系或把执行清单升级成新 Task。
- 子代理交接提示词锁定父提示词的完整开发计划和既定 Task；子代理只列本子任务执行清单，不自行重拟计划。
- 如果当前工作是继续优化已有提示词，改写结果保留了原有仍有效的硬规则、Task/step 边界、并行/串行约束、双审门和禁止项；没有被新文本整段覆盖。
- 最终验收集中执行，并明确拒收缩水交付。
