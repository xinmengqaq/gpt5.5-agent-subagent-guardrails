# 子代理交接模板

派发任何 GPT-5.x 子代理前必须读取本文件。未读取本文件，不得生成子代理交接提示词。

## 上下文定位与真源下发

目标是节省 token，并降低子代理误读全库、旧文档或无关资料的概率。

派发任何执行子代理前，主代理必须先判断当前 step 是否需要额外真源：

- 已有计划拆出 step/子步骤时，主代理必须先定位当前 Task 的 step 清单，至少掌握每个 step 的编号/名称、目标、依赖或等待关系、主要文件范围和验证/证据要求。
- 如果还没有掌握 step 清单，先用低成本上下文定位拿到精确行号和极短摘要；不得凭 Task 总目标猜测并行/串行，也不得直接派发 `Step 1-5`、`Step 1~5`、`步骤 1 到 5`、`完成前五步` 这类连续 step 范围。
- 当前用户消息、计划原文和主代理已读片段已经足够时，直接下发这些精准片段，不再派发只读定位任务。
- 需要补定位且主代理尚未掌握精确行号时，先用最低可用推理强度派发只读上下文定位子代理，默认 `gpt-5.4 low`；如果没有可派发的低成本子代理，主代理用 `rg`、`rg --files`、Graphify 索引或目标文件轻量读取完成定位。
- 上下文定位子代理只返回 `路径:起始行-结束行`、片段用途、极短摘要和仍缺的信息；不得写方案、做实现、做审查、跑全库扫描或替主代理判断产品行为。
- 定位必须从当前用户消息、真源计划、已知路径、Graphify 索引或明确关键词出发；不得让定位子代理从仓库根目录无目标扫全库。

子代理只拿到当前 step 需要的真源：

- 真源片段必须按需下发。当前 step 不需要 UI 原型、API 契约、状态卡、Graphify、日志或其他资料时，直接删除对应行，不写“无”。
- “真源片段包”只放任务事实、产品/代码契约、计划原文、状态证据、必要路径行号和执行必须读的代码/资料片段。
- 不得把主代理元决策写进“真源片段包”：技能选择/技能路由、主代理后续使用哪个技能、模型选择理由、并行上限判断、调度批次、读取纪律、防臆测规则、不要进入下一 Task、不要改哪些范围等，都属于交接字段或执行边界。
- “当前用户原始约束”只摘当前子任务必须遵守的产品、代码、范围或验收约束。用户关于主代理工作流、技能切换、提示词路由、模型/并行策略的决策不写入此项。
- 已经由主代理或上下文定位子代理读过的资料，不要求执行子代理重新读取全文；只给摘要、路径、行号和必要定位关键词。
- 执行子代理默认只读交接提示词列出的目标片段和少量相邻行；发现片段不足时先问主代理，不自行扩大到全库扫描。
- “可读”范围优先写精确文件片段，例如 `docs/plan.md:42-68`、`frontend/src/App.tsx:120-165`；只有真实需要跨文件搜索时，才写具体 `rg` 关键词和限定目录。
- 主代理内部的并行/串行判断默认不下发给子代理。只有子代理确实需要等待前置 step、避免共享写入冲突或遵守外部顺序门时，才保留一句最小约束；不写主代理的调度理由、批次解释或并行分析过程。

## 模板

空段落直接删除。不要把 GPT-5.5 风格说明写给子代理。

创建子代理时，工具调用的 `model` 字段必须与模板首行一致；只在提示词里写 `模型：gpt-5.3-codex` 或 `模型：gpt-5.4` 不会改变实际子代理模型。若工具调用缺 `model` 或与首行不一致，必须废弃该子代理并重新派发。

```markdown
模型：<gpt-5.3-codex|gpt-5.4|gpt-5.5>
推理强度：<gpt-5.3-codex 固定写 xhigh；gpt-5.5 只能写 low|medium|high；gpt-5.4 可写 low|medium|high|xhigh>
身份背景：<按子任务领域写具体专家身份；不得写成主代理、协调者、总负责人或总验收人>
轻量日志：按 `references/subagent-log.md` 追加到项目根目录 `subagent-log/YYYY-MM-DD-subagent-log.md`；并行写入前先重读当天文件，只追加自己的段落。日志不写内部推理链或模型选择理由，不能代替验收证据。
前置技能：<后端/前端逻辑编码实现/改代码/重构写“先启用 karpathy-guidelines”；UI/视觉/布局/图形/响应式/原型对齐实现写“先启用 web-design-engineer 和 playwright，并在实现过程中用真实浏览器迭代调试”；真实浏览器验证/浏览器验收/视觉验收/截图取证/响应式核对写“先启用 playwright”；满足多项就全部写；都不满足则删除本行>

所属 Task：
<Task 名称和目标状态；这里只写归属，不是交付边界>

子任务：
<当前 Task 内的单个 step/子步骤窄交付物；禁止写成“完成整个 Task”、`Step 1-5`、`Step 1~5`、`步骤 1 到 5`、`完成前五步`，也禁止覆盖实现、集成、验收、双审的端到端任务>

真源片段包：
- <只保留当前 step 必需项；不需要的项直接删除，不写“无”。>
- 当前用户原始约束：<只摘当前子任务必须遵守的产品、代码、范围或验收约束；不写技能路由、主代理工作流、模型/并行策略>
- 所属 Task 原文：<路径:起始行-结束行；摘录或极短摘要>
- 当前 step/子步骤原文：<路径:起始行-结束行；摘录或极短摘要>
- 相关计划/规格/状态卡：<路径:起始行-结束行；需要它的原因>
- UI 原型：<路径:起始行-结束行；需要保持的页面/组件>
- API/数据契约：<路径:起始行-结束行；需要遵守的字段/接口>
- Graphify 上下文：<仅在需要项目关系、模块边界、代码路径或上下文索引时写；给出具体 query/path/explain 结果摘要>
- 代码/资料片段：<路径:起始行-结束行；只给执行必须读的片段>

并行/串行：
<默认删除本段。只有当前子任务必须等待某个前置 step 完成、必须避开共享写入冲突、或必须遵守外部顺序门时才保留，并只写最小执行约束，例如“等待 Step 3 合并后再继续”或“不要改 X 文件，避免和并行 step 冲突”；不写主代理的调度解释、依赖分析、批次理由或并行判断过程>

范围：
- 可读：<精确文件片段优先，例如 `docs/plan.md:42-68`；确需搜索时写限定目录和 `rg` 关键词>
- 可写：<只在需要写文件时保留；写精确路径>
- 不要碰：<只在存在禁止触碰范围时保留>

完成标准：
- <当前 step/子步骤必须完整实现的产品级行为>
- <Task 层覆盖测试模型：需要统一测试覆盖视角时，使用 gpt-5.5 medium/high 输出 Task 级行为边界、失败路径、契约断点、状态组合、回归风险、覆盖矩阵和关键断言方向；不写 step 实现代码>
- <后端或前端逻辑 step：需要 TDD/测试先行时，当前执行子代理必须先按 Task 层覆盖测试模型里的相关设计实现自己的 step，包括实现代码、失败测试、回归测试、契约测试、单测/组件测试或少量 E2E spec，并跑对应测试；覆盖模型不足时，只在自己的 step 范围内补充实现与 TDD；不适用则删除本行>
- <必须覆盖的边界情况、错误/空态或响应式状态；不适用则删除本行>
- <涉及 UI/视觉/布局/图形/交互状态/响应式/原型对齐时，不强制 TDD；必须包含 `web-design-engineer` 设计实现判断，以及 `$playwright` CLI-first 实现期调试证据：fresh snapshot、截图路径、页面/状态/viewport、关键观察结果和根据截图做过的修正；已有 E2E spec 通过也不能删除本项>
- <可观察结果>
- <必需证据>

执行边界：
- 执行前先确认本交接提示词中的所属 Task、子任务、范围和完成标准。
- 不得自行重拟开发计划、重拆任务体系、改写作品目标或改变所属 Task/step。
- “所属 Task”只用于理解归属，不是交付边界；交付只覆盖“子任务”字段中的单个 step/子步骤。
- 如果“子任务”包含连续编号或多个 step 范围，例如 `Step 1-5`、`Step 1~5`、`步骤 1 到 5`、`完成前五步`，立即报告交接提示词违规并要求主代理按单个 step/子步骤重发。
- 不重新获取主代理已经给出的真源全文，不从仓库根目录无目标扫描；只读真源片段包和范围字段列出的目标片段。片段不足时先向主代理咨询，等待答复后继续。
- 禁止依据记忆、旧提示词、Graphify 或主观猜测补产品行为、提示词模板、技能路由或并行上限。
- 遇到需要跨 step、集成、双审、最终验收、范围变更或完成标准不清的内容，先向主代理咨询，等待答复后继续。
- 只允许列出本子任务执行清单；执行清单必须覆盖本子任务的真实产品行为、架构或数据影响、边界情况、错误处理、检查方式和交付证据。

质量底线：
- 不得提交失败交付。
- 必须完整交付“子任务”字段对应的当前 step/子步骤；如果该 step/子步骤本身范围不清或真源片段不足，先向主代理咨询，不得自行降级成只打通主路径。
- 后端和前端逻辑子任务按风险使用 TDD/测试先行；`gpt-5.5 medium/high` 只在整个 Task 需要统一覆盖测试模型时使用，step 执行代理按该覆盖模型实现自己的 step，覆盖不足时再在 step 范围内补充实现与 TDD。UI、视觉、布局、图形、响应式和原型对齐子任务不强制 TDD，必须改用 `web-design-engineer` + `playwright` 实现期浏览器迭代。
- 前端子任务不得破坏已定原型 UI；只能把硬编码数据替换为真实数据流、接口、状态或配置，并用真实浏览器验证受影响路径。
- 涉及真实浏览器验证、浏览器验收、视觉验收、页面交互复现、响应式核对或截图取证时，必须先启用 `playwright`，并按该技能的 CLI-first 方式执行；不得跳过技能直接口头声称“已浏览器验证”，也不得默认改写成 Playwright 测试文件，除非父提示词明确要求生成测试代码。
- UI/视觉相关 step 不能只交 E2E spec、`@playwright/test`、DOM 断言、组件快照、静态扫描或测试报告；这些只能作为流程证据，不能替代 `$playwright` CLI snapshot/screenshot 的视觉证据。
- 如果主代理验收判定命中失败交付，必须按原 Task/step 返工；不得要求主代理接管实现，不得用 TODO、说明文字或验收降级替代返工。
- 真源片段不足、UI 原型路径不明确、step 完成标准不清、需要跨 step/跨 Task/共享架构决策、可能破坏原型 UI、改变范围、面对高影响方案选择、降低验收标准或做复杂架构判断前，先向主代理咨询；主代理只按当前用户消息、真源计划、原型、代码、规格或状态卡直接答复，不接管执行，也不会另派顾问子代理。
- Graphify 更新按变更类型触发：后端代码、前端逻辑、共享类型/契约、API/数据模型、架构相关文件、关键配置、影响模块关系的重要文档必须运行 `graphify update .`；纯 UI/样式/视觉微调、纯文案、截图产物、日志、只读审查、纯验收、纯截图取证可不运行，但必须说明理由。如果应运行但失败，返回失败命令、关键错误和已尝试的低风险处理。

输出：
- 改了什么或发现了什么。
- 证据：命令、测试、截图、文件路径、日志、失败信息或阻塞项。
- 证据：需要 Graphify 更新的 step 必须包含 `graphify update .` 结果、失败信息或阻塞项；不需要更新的 step 写明跳过理由。
- 日志：写明已追加的 `subagent-log/YYYY-MM-DD-subagent-log.md` 路径；如果无法写入，说明原因并把应追加的日志段落放在输出里。
- 是否满足完成标准。
```
