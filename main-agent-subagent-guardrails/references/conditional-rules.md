# 条件规则

涉及模型推理、UI/浏览器、Graphify、日志或前置技能时必须读取本文件。未读取本文件，不得生成相关交接提示词。

## 模型与推理强度

先选模型，再定推理强度。模型选择兼顾成本和风险，默认不用最贵模型。

可用子代理模型只有 `gpt-5.5`、`gpt-5.4` 和 `gpt-5.3-codex`；不得编造其他模型名。用户写 `gpt5.4` 时统一规范为 `gpt-5.4`，用户写 `5.3-codex` 或 `gpt5.3-codex` 时统一规范为 `gpt-5.3-codex`。

模型选择必须在创建子代理的工具调用里真实传递：

- 交接提示词首行写 `模型：gpt-5.3-codex` 时，创建子代理工具调用必须显式传 `model: "gpt-5.3-codex"`。
- 交接提示词首行写 `模型：gpt-5.4` 时，创建子代理工具调用必须显式传 `model: "gpt-5.4"`。
- 交接提示词首行写 `模型：gpt-5.5` 时，创建子代理工具调用必须显式传 `model: "gpt-5.5"`。
- 只在提示词正文里写模型、不在工具调用里传 `model`，属于模型分配违规，因为实际执行会落到系统默认模型。
- 工具调用里的 `model` 与交接提示词模型不一致时，必须废弃该子代理并按一致模型重新派发。

`gpt-5.3-codex` 适用：

- 纯编程密集型任务：真源清楚、目标明确、主要是代码搬运/补齐/局部实现/按既定测试设计补测试，不需要产品判断、架构判断、覆盖策略判断或审美判断。
- 实时结对编程任务：主代理已给出精确片段、路径、边界和完成标准，子代理只负责窄范围执行、反馈阻塞和提交证据。
- 预算敏感编程任务：低风险、可回滚、可由测试或 diff 快速验证的小实现、小返工、小修补。
- 标准代码审查、规格符合性第一轮审查、轻量只读核对。
- 小范围 bugfix、简单测试补充、返工里的小改。
- 机械迁移、重命名、类型补全、局部后端/前端逻辑修正。
- 低风险窄范围 UI 验收、截图/viewport/状态证据核对、diff 范围核对。
- 前提是真源片段精确、可写范围明确、完成标准清楚，不需要它自行扫库、定方案或做高影响判断。

`gpt-5.4` 适用：

- 标准/重要审查、验收、重要集成审查、中等风险代码审查、困难问题定位。
- `gpt-5.3-codex` 不适合的中等风险判断型任务，或一次返工仍不收敛的实现任务。
- 中等返工单审：只检查原 finding 是否关闭、返工 diff 是否引入新增风险、相关测试或浏览器证据是否足够；不重审完整 Task。
- UI 视觉验收、截图证据核对、较复杂交互验收。
- 不作为普通实现默认模型；普通实现优先看是否可由 `gpt-5.3-codex xhigh` 承接。

`gpt-5.5` 适用：

- 高不确定性复杂实现。
- 含 TDD 的测试设计 step：负责行为边界、失败路径、契约断点、状态组合、回归风险、测试清单、覆盖矩阵和关键断言设计。
- 强耦合跨模块重构。
- 深层疑难调试。
- 发布阻断级修复。
- 高风险返工：涉及架构、真实 LLM 链路、数据模型、权限/安全、跨模块状态、发布阻断、多轮返工仍不收敛或可能改变产品行为。
- UI 实现、视觉改动、布局调整、响应式修正、图形/可视化实现、原型对齐改动。
- `gpt-5.4` 多轮返工仍不收敛的复杂执行子任务。

UI 路由硬规则：

- UI 实现/视觉改动/布局调整/响应式修正/图形或可视化实现/原型对齐改动：固定使用 `gpt-5.5 medium`。
- UI 验收/视觉审查/截图证据核对/viewport 状态核对：可用 `gpt-5.4 medium`；复杂交互验收或重要 UI 验收可用 `gpt-5.4 high`。
- 低风险、窄范围、只读的 UI 验收、截图/viewport/状态证据核对、diff 范围核对：可用 `gpt-5.3-codex xhigh`。
- `gpt-5.3-codex` 禁止做 UI 实现或视觉改动。
- `gpt-5.4` 禁止做主要 UI 实现或视觉改动；只有非常小的非视觉前端逻辑或机械样式修正，且不涉及审美/布局取舍/原型对齐时才可使用。

推理强度：

- `gpt-5.5 low`：范围明确的小块复杂执行。
- `gpt-5.5 medium`：中等规模复杂实现、复杂但边界清楚的重构。
- `gpt-5.5 high`：强耦合跨模块重构、深层疑难调试、关键链路复杂修复、发布阻断级修复、低强度多轮返工仍未收敛的问题、需要长链路推演和高稳定性的复杂执行子任务。
- `gpt-5.3-codex xhigh`：`gpt-5.3-codex` 一律使用最高推理强度，承接低风险窄任务、纯编程密集型任务、实时结对编程任务、预算敏感编程任务、标准审查、小改、小返工、简单测试补充、机械迁移、轻量核对；不得为省 token 降低推理强度。
- `gpt-5.4 low`：窄范围只读扫描、轻量信息核对、低风险材料整理、简单 follow-up。
- `gpt-5.4 medium`：标准代码审查、常规调试验收、UI 视觉验收、截图证据核对、中等风险审查。
- `gpt-5.4 high`：跨模块联动审查、复杂交互验收、困难问题定位、重要集成审查、重要 UI 验收。
- `gpt-5.4 xhigh`：高不确定性复杂实现、发布阻断级修复、低强度多轮返工仍未解决的问题。

禁止：

- 不得把所有任务都默认交给 `gpt-5.5`。
- 不得把普通低风险实现默认交给 `gpt-5.4`；先判断是否可由 `gpt-5.3-codex xhigh` 承接。
- 不得把复杂任务下压给 `gpt-5.3-codex` 只为省 token。
- 不得把含 TDD 的测试设计 step 派给低模型；低模型只能按高模型给出的测试清单实现测试，不得自行缩减、改写或扩大覆盖策略。
- `gpt-5.3-codex` 一律写 `xhigh`；不得使用 low/medium/high。
- 不得把 UI 实现、视觉改动、布局调整、响应式修正、图形/可视化实现或原型对齐改动派给 `gpt-5.3-codex` 或 `gpt-5.4`。
- 不得为了“保险”把所有 `gpt-5.4` 审查、验收、只读扫描都写成 `xhigh`。
- 不得把所有返工默认升级成完整双审；必须先按小返工自验、中等返工单审、高风险返工双审分流。
- 不得让返工复审重新读取完整 Task、完整计划或全库；复审只看原 finding、返工 diff、必要相邻行、相关测试或浏览器证据。
- `gpt-5.5` 不启用 `xhigh`；原本应归入更高强度的任务统一使用 `gpt-5.5 high`。
- `gpt-5.3-codex` 不做架构判断、跨模块重构、高风险数据/状态流改动、UI 实现、视觉改动、复杂 UI 调整、真实 LLM 链路问题、发布阻断级修复、多轮返工仍不收敛的问题或最终集中验收。
- 主代理内部自行判断模型选择理由，不要把模型理由写进子代理交接提示词。

## 前置技能

- 编码实现、改代码或重构子代理：必须先启用 `karpathy-guidelines`。
- 涉及 UI 实现、前端视觉调整、截图后调整、视觉审查、设计系统/UI Kit 对齐、交互细节调整或响应式视觉修正：必须先启用 `web-design-engineer`。
- 涉及真实浏览器验证、浏览器验收、浏览器调试、页面交互复现、截图取证：必须先启用 `playwright`。
- 同时满足多类条件时，相关技能都写。

## 前端/UI 与浏览器

## 测试策略分层

子代理派发时必须按工作层次选择验证方式，不得把所有前端工作都套成同一种 TDD 流程。

含 TDD 的任务按 step 分工：

- 测试设计 step 使用 `gpt-5.5 medium/high`：负责识别行为边界、失败路径、契约断点、状态组合、回归风险，产出测试清单、覆盖矩阵和关键断言。范围清楚时用 `medium`，跨模块/真实 LLM/复杂状态机/高风险契约用 `high`。
- 测试实现 step 使用 `gpt-5.3-codex xhigh`：只按测试设计清单补测试文件、断言、fixture、mock、回归用例并运行对应测试；不得自行缩减、改写或扩大测试设计。
- 如果测试实现发现设计清单不可执行、与当前代码冲突或缺关键真源，先咨询主代理，不自行改测试策略。

后端适合 TDD/测试先行：

- API 行为、provider/adapter、数据模型、权限、错误处理、边界条件、持久化、状态推进、异步任务。
- 交接提示词可以要求先补失败测试、回归测试或契约测试，再实现并跑对应后端测试。

前端逻辑适合轻量 TDD：

- 数据映射、store/state machine、loading/empty/error/ready 状态、表单校验、路由守卫、关键交互后的状态变化、前后端契约适配。
- 用单测、组件测试或少量 E2E spec 覆盖行为确定性；E2E spec 只证明流程和状态，不证明视觉质量。

UI/图形层不强制 TDD：

- 页面布局、视觉层级、间距、字体、颜色、动效、响应式、原型对齐、图形化组件、画布、可视化。
- 不要求先写视觉测试，不默认用像素快照或 E2E spec 锁死视觉细节。
- UI/图形实现子代理必须同时启用 `web-design-engineer` 和 `playwright`，在实现过程中用真实浏览器迭代：实现一小块 -> 打开页面 -> snapshot/screenshot -> 对照原型或目标状态调整 -> 再截图确认。
- 输出必须包含实现中浏览器调试证据：命令、页面/路由、状态、viewport、截图路径/产物位置、关键观察结果，以及根据截图做过的修正。
- 独立视觉验收仍保留，但主要检查实现子代理是否已经给出足够浏览器证据和原型对齐结果，不重复完整开发调试。

不要扩大成本：

- 纯后端不触发 `web-design-engineer` 或浏览器。
- 纯前端逻辑不强制视觉截图，除非影响页面呈现。
- UI 截图按影响分级，不做全页面全状态穷举，除非当前用户或计划明确要求。

截图分级：

- 小 UI 改动：1 个核心 viewport + 1 个关键状态截图；适用于文案层级、局部样式、单组件轻微布局调整。
- 中等 UI 改动：桌面 + 移动各 1 个关键状态；如果有 hover/open/error/empty 等明显受影响状态，再补 1-2 张。
- 大 UI、响应式、图形/可视化、导航结构或多组件联动改动：覆盖核心路径、关键状态、错误/空态和目标 viewport；仍以受影响范围为边界，不做无关页面全量截图。

每个前端 Task 都必须包含真实浏览器验证证据。不得只用静态扫描、单元测试、组件快照、mock 数据或主观描述替代浏览器实测。

真实浏览器验证必须覆盖该 Task 影响到的核心用户路径、关键状态、错误/空态和响应式边界。

涉及 UI、视觉、布局、交互状态、响应式或原型对齐的 step，E2E spec 通过只算流程/断言证据，不算视觉验收完成。必须追加 `$playwright` CLI-first 真实浏览器核对，至少包含 fresh snapshot、关键状态截图、受影响路径截图；有响应式要求时还要覆盖目标 viewport。

涉及真实浏览器验证的子代理交接提示词必须明确要求按 `playwright` 技能执行真实浏览器 CLI 流程，默认走 skill 内的 wrapper/workflow，不改写成 Playwright 测试文件，除非当前用户明确要求生成测试代码。

禁止：

- 不得用 `@playwright/test`、E2E spec、DOM 断言、组件快照或测试报告替代视觉截图取证。
- 不得因为已有 E2E spec 通过就跳过 `$playwright` 的 CLI snapshot/screenshot 流程。
- 不得只说“视觉正常”“浏览器验证通过”；必须返回命令、页面/路由、viewport 或状态、截图路径/产物位置和关键观察结果。

前端实现必须保护已经定好的原型 UI：

- 不得重做视觉风格。
- 不得重做布局。
- 不得重做交互结构。
- 不得重做文案层级。
- 不得重做组件形态。
- 只能把硬编码数据替换为真实数据流、接口、状态或配置。
- 不得以“去硬编码”为理由破坏原型 UI。

## Graphify

Graphify 只能用于项目文件关系、模块边界、代码路径和上下文索引。不得把 Graphify 结果当成产品行为、提示词模板、技能路由或并行上限的真源。

可以使用：

- `graphify-out/GRAPH_REPORT.md`
- `graphify query`
- `graphify path`
- `graphify explain`

Graphify 更新按变更类型触发，避免无意义成本：

- 必须运行 `graphify update .`：后端代码、前端逻辑、共享类型/契约、API/数据模型、架构相关文件、关键配置、影响模块关系的重要文档。
- 可不运行：纯 UI/样式/视觉微调、纯文案、截图产物、日志、只读审查、纯验收、纯截图取证，且没有改变代码依赖、模块关系、API 契约或业务逻辑。
- 如果不运行，输出中写明“不运行 graphify update”的理由和变更类型。
- 如果应运行但命令失败，必须返回失败命令、关键错误和已尝试的低风险处理，不得隐瞒或跳过。

## 日志

轻量日志只记录事实索引和证据，不反向要求子代理读取全部日志历史；并行追加前只重读当天日志文件。

详细格式见 `references/subagent-log.md`。日志不是验收证据本体，不能代替测试、浏览器截图、命令输出或审查结论。
