# 质量门

Task 验收、双审、集中验收或最终验收前必须读取本文件。未读取本文件，不得宣布合格或完成。

按阶段检查。不要每个阶段都展开全量清单；只检查当前阶段相关项。最终验收必须覆盖全部阶段。

## 阶段 A：主代理合同检查

- 本技能被直接当作 GPT-5.5 主代理提示词执行，没有再为主代理重写一份合同。
- 没有使用旧块名：`运行时：codex`、`上下文`、`工具规则`、`验证`。
- GPT-5.5 风格只出现在主代理提示词组织和主代理质量门里；子代理交接提示词没有写“按 GPT-5.5 风格执行”“结果优先提示词结构”或类似风格迁移要求。
- 没有漏掉本技能中的主代理职责边界、真源优先级、模型分配、子代理调度、验收门、返工规则、证据标准、前端保护约束和停止条件。
- 记忆只作为线索，没有覆盖当前用户消息、当前真源或本技能。
- Graphify 只作为项目关系、模块边界、代码路径和上下文索引线索，没有覆盖本技能的提示词形状、技能路由、并行规则或产品行为。

## 阶段 B：派发与交接提示词检查

- 主代理派发前已经完成低成本上下文定位；子代理拿到的是路径、行号、摘要和必要关键词，不需要重新全文读取主代理已读资料。
- 子代理交接提示词按需下发真源；不需要的 UI 原型、API 契约、状态卡、Graphify、日志等字段已经删除，没有堆叠“无”造成噪音。
- “真源片段包”只包含当前子任务需要的任务事实、产品/代码契约、计划原文、状态证据、必要路径行号和执行必须读的代码/资料片段。
- “真源片段包”没有混入主代理元决策：技能选择/技能路由、主代理后续使用哪个技能、模型选择理由、并行上限判断、调度批次、读取纪律、防臆测规则、不要进入下一 Task、不要改哪些范围等。
- “当前用户原始约束”只保留当前子任务必须遵守的产品、代码、范围或验收约束；没有把用户对主代理工作流、技能切换、提示词路由、模型/并行策略的决策当成真源。
- 如果真源计划已拆 Task，沿用计划原文 Task，没有重新识别 Task 边界、重新设计任务体系或把执行清单升级成新 Task。
- 如果真源计划已拆 step/子步骤，主代理派发前已经读取或定位当前 Task 的 step 清单，并掌握每个 step 的编号/名称、目标、依赖或等待关系、主要文件范围和验证/证据要求。
- 主代理没有在未读取 step 清单时凭 Task 总目标猜测并行/串行，也没有直接生成“完成 Step 1-5”之类交接提示词。
- 如果计划没有写明 step/子步骤的并行/串行，主代理已经按依赖、写入范围、共享决策、验证顺序和审查门评判可并行或必须串行的 step；该评判只用于派发批次，没有改变 Task/step 边界或验收口径。
- 当前 Task 按实施计划 step/子步骤派发子代理推进，没有把整个 Task 包给单个子代理端到端开发。
- 每个子代理交接提示词的“所属 Task”和“子任务”不是同一粒度；“所属 Task”只标归属，“子任务”是可独立验收的 step/子步骤。
- 每个“子任务”都只包含单个 step/子步骤；没有 `Step 1-5`、`Step 1~5`、`步骤 1 到 5`、`完成前五步`、`Step 1/2/3 一起完成` 等连续编号或多 step 范围。
- 每个子代理交接提示词都有“真源片段包”，并只写当前 step 必需的当前用户产品/代码/范围/验收约束、Task/step 原文、相关计划/规格/状态卡、UI 原型、API/数据契约、Graphify 或代码片段；每项尽量有路径和行号。
- 子代理交接提示词允许用 `graphify-out/GRAPH_REPORT.md`、`graphify query`、`graphify path`、`graphify explain` 找项目关系和上下文索引，并禁止让子代理依据记忆、旧提示词、Graphify 或主观猜测补产品行为、提示词模板、技能路由或并行上限。
- 子代理交接提示词默认没有“并行/串行”解释段；主代理内部的调度理由、批次解释、依赖分析和并行判断过程没有下发给子代理。
- 只有当前子任务必须等待前置 step、避免共享写入冲突或遵守外部顺序门时，交接提示词才保留一句最小化并行/串行约束；没有额外塞入主代理视角的调度说明。
- 没有整 Task 外包表述。
- 没有连续 Step 打包表述。
- 没有把失败交付写成可接受口径。
- 集成、真实浏览器验收、真实 LLM 验收、必要审查和集中验收没有和功能实现一起打包给同一个子代理。
- 子代理并行上限默认 4；只有当前用户消息或当前真源文件明确写更低上限时才降低。
- 并行子任务写入范围互不重叠，或都是只读；必须串行的依赖、共享写入、共享决策、集成、真实 LLM 验收、真实浏览器验收、双审和集中验收没有被并行派发。
- 模型选择不是一律 `gpt-5.5`；`gpt-5.3-codex`、`gpt-5.4` 和 `gpt-5.5` 都按风险写明推理强度，且审查/验收/只读扫描没有被无理由抬到 `xhigh`。
- `gpt-5.3-codex` 用于真源片段精确、可写范围明确、完成标准清楚的纯编程密集型任务、实时结对编程任务、预算敏感编程任务、低风险审查、小实现、小返工、简单测试补充、机械迁移或轻量核对；没有承担架构判断、跨模块重构、高风险数据/状态流改动、复杂 UI 调整、真实 LLM 链路问题、发布阻断级修复、多轮不收敛问题或最终集中验收。
- `gpt-5.3-codex` 的推理强度固定为 `xhigh`；没有写成 low、medium 或 high。
- 普通低风险实现没有默认派给 `gpt-5.4`；主代理已优先判断是否可由 `gpt-5.3-codex xhigh` 承接。`gpt-5.4` 更多用于审查、验收、集成审查、中等风险代码审查、困难问题定位和较复杂交互验收。
- UI 实现、视觉改动、布局调整、响应式修正、图形/可视化实现和原型对齐改动固定派给 `gpt-5.5 medium`；没有派给 `gpt-5.3-codex` 或 `gpt-5.4`。
- UI 验收、视觉审查、截图证据核对和 viewport 状态核对可派给 `gpt-5.4 medium/high` 或低风险窄范围的 `gpt-5.3-codex xhigh`；验收模型没有接管 UI 实现。
- 每个子代理的交接提示词首行 `模型：...` 与创建子代理工具调用里的 `model` 字段完全一致；没有只在提示词里写模型却漏传工具层 `model` 字段。
- 创建子代理工具调用缺 `model` 字段、模型名拼写不规范或与交接提示词不一致时，已经废弃错误派发并按正确模型重新创建。
- `gpt-5.5` 不使用 `xhigh`；原本应归入更高强度的任务已经下放到 `gpt-5.5 high`。
- 编码实现子代理的交接提示词写清了先启用 `karpathy-guidelines`；纯审查、只读扫描、验收或调度子代理没有被误要求启用，除非它们也要写代码。
- 后端和前端逻辑子代理按风险写清 TDD/测试先行要求；UI、视觉、布局、图形、响应式和原型对齐子代理没有被误要求先写视觉测试或用 E2E spec 锁死视觉细节。
- `gpt-5.5 medium/high` 只在整个 Task 需要统一覆盖测试模型时使用，输出 Task 级行为边界、失败路径、契约断点、状态组合、回归风险、覆盖矩阵和关键断言方向；没有要求它介入每个 step 的 TDD 或实现。
- step 执行代理按 Task 层覆盖测试模型实现自己的 step，包括实现代码、测试和证据；覆盖模型不足时，只在自己的 step 范围内补充实现与 TDD，没有把普通 step 默认升级给 `gpt-5.5`。
- 涉及 UI 的实现、截图调整或视觉审查子代理写清了先启用 `web-design-engineer`；非视觉逻辑、后端、CLI、数据处理或纯文本审查没有被误要求启用。
- 涉及 UI/视觉/布局/图形/响应式/原型对齐的实现子代理写清了当前子任务实际需要的前置技能，例如 `web-design-engineer`、`Playwright CLI Skill` 或其他命中的前置技能，并要求实现过程中用真实浏览器迭代调试，而不是只在完成后做一次浏览器验收。
- 涉及真实浏览器验证、浏览器验收、视觉验收、页面交互复现、响应式核对或截图取证的子代理写清了先找并启用名为 `Playwright CLI Skill` 的技能；没有把真实浏览器验证只写成模糊要求，也没有写成自行编写脚本触发 Playwright。
- 子代理交接提示词没有把前置技能降级成只写 Playwright；编码、UI/视觉和浏览器验证同时命中时，`karpathy-guidelines`、`web-design-engineer` 和 `Playwright CLI Skill` 按需全部下发。
- 每个子代理交接提示词都写清身份背景，且身份背景与子任务领域匹配。
- 每个子代理交接提示词都写清轻量日志规则，并引用 `references/subagent-log.md`；日志目录是项目根目录 `subagent-log/`，日志文件按日期新建或追加。
- 日志规则明确禁止内部推理链、模型选择理由和长篇过程；日志只作为执行索引，不能替代验收证据。
- Graphify 更新按变更类型触发：后端代码、前端逻辑、共享类型/契约、API/数据模型、架构相关文件、关键配置、影响模块关系的重要文档要求运行 `graphify update .`；纯 UI/样式/视觉微调、纯文案、截图产物、日志、只读审查、纯验收、纯截图取证没有被误要求运行。
- 需要 Graphify 更新的 step 已把 `graphify update .` 结果纳入证据；跳过更新的 step 已写明变更类型和跳过理由。
- 子代理咨询链路只允许“子代理 -> 主代理 -> 子代理”；主代理直接依据真源答复，不另派顾问子代理。

## 阶段 C：Task 验收检查

- 主代理验收职责写清“合格 / 返工 / 阻塞”判定；命中失败交付必须退回子代理返工，不能进入双审或最终验收。
- 主代理验收必须检查子代理是否使用真源片段、完整覆盖当前 step/子步骤、没有擅自臆测不确定项、没有命中失败交付；不满足则返工。

## 阶段 D：UI / 浏览器条件检查

- 前端 Task 写入真实浏览器验证要求，且不把静态扫描、快照、mock 或描述当作浏览器实测。
- 真实浏览器验证相关子代理交接提示词明确走 `Playwright CLI Skill`，且沿用其 CLI-first 约束；没有被错误改写成默认生成 `@playwright/test` 测试文件或自行编写 Playwright 脚本。
- 涉及 UI、视觉、布局、动效、质感、交互状态、响应式或原型对齐的 step，即使已有 E2E spec 通过，也有原型源码层核对、`Playwright CLI Skill` CLI-first 的 fresh snapshot、截图路径/产物、页面/状态/viewport 和关键观察结果。
- UI/浏览器证据同时覆盖原型源码层和浏览器呈现层；只有截图、只有 DOM/E2E、只有口头描述或只查看 HTML 静态视觉都不合格。
- UI 原型对齐不能只复刻静态 HTML 视觉。若遗漏原型源码中的 animation、transition、状态触发、阴影、模糊、透明度、渐变、层次质感或交互细节，判定为原型对齐失败，必须返工。
- UI/图形层验收依据是真实浏览器调试证据和原型/目标对齐；没有把 TDD、像素快照、E2E spec 或 DOM 断言当成默认视觉门槛。
- UI 实现子代理输出包含实现中浏览器调试证据，以及根据截图或 snapshot 做过的修正；独立视觉验收只做确认/抽查，不重复完整开发调试。
- UI 截图按影响分级：小 UI 改动只需 1 个核心 viewport + 1 个关键状态；中等 UI 改动覆盖桌面 + 移动和受影响关键状态；大 UI/响应式/图形/多组件联动才覆盖核心路径、关键状态、错误/空态和目标 viewport。
- 没有把 `@playwright/test`、E2E spec、DOM 断言、组件快照、静态扫描或测试报告当成视觉验收完成证据。
- 前端 Task 写入原型 UI 保护约束：不重做既定视觉、布局、交互、文案层级或组件形态，只替换硬编码数据来源。
- UI 子代理必须拿到原型路径和需要保持的页面/组件；没有原型对照证据的 UI 交付直接返工。

## 阶段 E：审查与最终验收检查

- 必要审查由审查子代理执行，且只在当前 Task 的 step/子步骤开发、集成和必需验收证据完成后启动。
- 审查先做规格符合性审查；只有实际实现或修改了后端、前端逻辑、共享契约、状态机、数据流、测试或可执行逻辑的 Task/子任务才追加代码质量审查；纯 UI/视觉/布局/样式/响应式调优且没有改变前端逻辑、数据流、状态机、API 契约或业务行为时，不走双审，只走 `Playwright CLI Skill` 视觉验收和必要 diff 范围核对。
- 返工复审没有默认重跑完整双审；已按风险分流：小返工由主代理核对返工 diff、原 finding 闭环和相关测试证据；中等返工只做一次单审；高风险返工或多轮不收敛才恢复双审。
- 小返工范围只包含测试、类型、命名、边界判断、单文件小补丁、机械修复或低风险局部补漏；没有把核心逻辑、API 契约、状态流、跨多文件改动或产品行为变化伪装成小返工。
- 中等返工单审提示词只包含原 finding、返工 diff 文件/行号、完成标准、相关测试或浏览器证据；没有重发完整 Task、完整计划、技能路由、并行/串行解释或无关真源。
- 高风险返工涉及架构、真实 LLM 链路、数据模型、权限/安全、跨模块状态、发布阻断、多轮返工仍不收敛或产品行为变化时，已恢复必要双审。
- 纯 UI/视觉/布局/样式/响应式返工且没有改变前端逻辑、数据流、状态机、API 契约或业务行为时，没有走返工双审；已走 `Playwright CLI Skill` 视觉验收、截图/viewport 证据和必要 diff 范围核对。
- 前端逻辑变更仍要求测试/TDD 和必要审查；没有把数据映射、状态机、表单校验、路由守卫、交互状态变化伪装成纯 UI 调整来跳过审查。
- 当前 Task 完成为产品级切片后才开始审查。
- 当前计划 Task 完成 step/子步骤开发、集成、必需验收证据、必要审查和集中验收后，才开始下一个计划 Task。
- 如果当前工作是继续优化已有提示词，保留原有仍有效的硬规则、Task/step 边界、并行/串行约束、双审门和禁止项，没有被新文本整段覆盖。
- 最终验收集中执行，并明确拒收失败交付。
